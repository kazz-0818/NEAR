# AI Scope Boundary（GPT寄り運用）

## 目的

この文書は、NEAR の運用を「GPT寄り」で進めるための責務境界を定義します。  
ここでの「GPT寄り」は、**チャットによる思考支援・文章生成・要件整理を主用途にする**ことを指します。

一方で、再現性・安全性・監査性が必要な挙動は、必ず開発要件として実装対象にします。

## 境界定義（先に結論）

### GPT寄りで扱う範囲（チャットで完結してよい）

- 文章の下書き、要約、言い換え
- 論点整理、選択肢比較、意思決定の補助
- 要件の叩き台、受け入れ条件の草案
- 疑似コードや実装案の提案

### 開発要件に必ず入れる範囲（GPT単体に任せない）

- 認証・認可、権限モデル
- データ保存・削除・バックアップ方針
- 外部 API 連携（再試行、失敗時挙動、レート制限）
- 監査ログ、追跡可能性、運用時の説明責任
- 可用性対策（冪等性、ロールバック、障害復旧）
- 性能・コスト上限と計測基準
- 法務・コンプライアンスに関わる制御

## 分岐ルール（運用時の判断）

新しい依頼は、次の順で判定します。

1. 一度きりの説明・草案で足りるなら GPT寄りで処理する。
2. ユーザー操作に対して毎回同じ挙動が必要なら開発要件にする。
3. 秘密情報・個人情報・権限が絡むなら開発要件にする。
4. ミス許容が低い業務なら、テストとガードレール込みで開発要件にする。

## 要件記述ルール

受け入れ条件を「AIが説明できること」で書かない。  
次の実行可能な要素で記述する。

- 誰が実行するか（actor）
- どこから実行するか（API / UI / Job）
- 入力と出力
- 認可制約
- エラー時挙動と再試行ポリシー
- 観測可能性（ログ / メトリクス）

## そのまま使えるテンプレート

### GPT寄りタスク（非本番）

- 目的: 文面・選択肢・実装候補の整理
- 制約: 本番データや本番状態を直接変更しない
- 成果物: ドラフト、提案、検討メモ

### 開発要件タスク（本番）

- トリガー: ユーザー操作またはシステムイベントを明示
- 挙動: 再現可能でテスト可能なシステム応答
- セキュリティ: AuthN/AuthZ と秘密情報管理を定義
- 運用: 監視、復旧、ロールバック手順を定義
- SLO/コスト: 遅延・エラー率・コスト上限を定義

## チーム運用ポリシー

- GPTは思考と設計の加速に使う。
- 本番挙動の責任はコード・インフラ・運用設計が持つ。
- 迷ったら開発要件に倒す。
